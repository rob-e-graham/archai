## What changed this pass

### 1. Auckland images fixed (was: all broken on the live site)

Root cause: all 120 Auckland records had `http://api.aucklandmuseum.com/...`
image URLs. On the https website these are blocked as **mixed content**, so
every Auckland image failed to render. The Auckland API fully supports https
(returns real 21–34KB images).

- `backend-archai/scripts/aucklandmuseum-harvester.js` — `mediaUrl()` now forces
  `http://` → `https://` so future harvests are clean.
- Live Qdrant: all 120 `archai_auckland` payloads patched in place
  (media_thumbnail / media_medium / media_large → https). Verified http=0, https=120.
- AUX.IO static pages regenerated, so the baked-in URLs are also https now.

Website search reads live Qdrant, so Auckland thumbnails are fixed immediately.

### 2. M+ rebuild gap closed

M+ (120 objects) was live in Qdrant but `archai_curator` was stale at 1205
(missing exactly M+'s 120), and AUX.IO had no M+ pages.

- Rebuilt curator via `POST /api/proxy/curator/build` → **1325 objects, all 10 sources**.
- Regenerated AUX.IO → **1191 pages** including 120 M+ pages.

The website "Collection Intelligence" / curator chat now searches M+ too.

### 3. Website — ACMI background line

`fineartmedia.tech` index.html: ACMI line now reads
"Exhibitions, Public Programs, AV & Cinema Tech — ACMI" (Public Programs was missing).
Committed + pushed.

## Brasiliana + ToMuCo — VERIFIED, NOT LIVE (decision needed)

Both harvesters Codex wrote work. Neither was onboarded into the live stack,
deliberately — each has an open issue that's a judgment call for Rob, and
dropping thin/legally-ambiguous data into a sovereignty-framed demo would
undercut the project. Dry-run results below.

### ToMuCo (Tokyo Museum Collection) — `scripts/tomuco-harvester.js`

- **Works.** Dry-run: **37 suitable records** (image-backed, bilingual JA/EN),
  real museums (Tokyo Metropolitan Art Museum, Teien Art Museum, Edo-Tokyo
  Open Air Architectural Museum). 1002 skipped for no image.
- **Legal flag:** every record shows "License URI not supplied." The harvester
  keeps them under mode `open_data_item_license`, relying on ToMuCo's general
  API terms (https://museumcollection.tokyo/en/terms/) rather than per-item
  licenses. Defensible for open data, but should be a *deliberate* call.
- **Volume:** ~37 is a smaller slice than the other sources (120 each).
- **To onboard:** confirm the terms are acceptable, then:
  ```
  node scripts/tomuco-harvester.js --limit 120
  ```
  (will likely land ~37 with current keyword/museum strategy)

### Brasiliana Museus — `scripts/brasiliana-harvester.js`

- **Works and is legally clean.** Dry-run: **6 records**, all "Domínio público"
  (public domain) with explicit rights URL. Correctly skips rights-restricted
  (160 skipped on rights, 84 on no image). Tainacan/WordPress API confirmed
  exposing structured metadata + direct images + reproduction conditions.
- **Volume problem:** current 10-keyword strategy yields only ~6 image-backed
  public-domain records. Needs a broader crawl (more keywords, or paginate the
  full public-domain set) to reach meaningful volume.
- **To onboard:** broaden the keyword/crawl strategy first, then:
  ```
  node scripts/brasiliana-harvester.js --limit 120
  ```
