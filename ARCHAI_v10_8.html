Connect a Collection — Harvester / API Setup
Point ARCHAI at your collection API. This builds a harvester config for the onboarding pipeline: raw records are preserved exactly as received, rights and cultural-protocol metadata stay attached per item, and nothing is published until it passes the legal and media gates. Your collection system is never modified — ARCHAI reads, it does not write.
1 · Institution
2 · Source
Keys are never stored in this page or the config — they live in the host keychain (Keytec) and are injected at harvest time.
3 · Rights & cultural safety defaults
What happens next
1
Download your connector file — it captures everything you entered here. Nothing has touched your systems; this is just a description of how ARCHAI should read them.
2
Send it to your IT contact — or to us. A test run reads a small sample of records and produces a report: what was found, what the rights gates would hold back, what the images look like. Read-only, reversible, nothing published.
3
Review together, then go live. After your team approves the report and the rights review, the full collection is connected. Your staff search it here; nothing appears publicly until you curate and publish it.
The full path: connect → test run → legal review → live connection → staff search → curated AUXIO publish. Questions at any step: rob@fineartmedia.tech
▸ Technical configuration — for your IT team
This connector drops into backend-archai/scripts/collection-targets.json, the registry the legal-harvest bot reads. Runs follow the standard gates: dry-run → item-rights check → media audit → curator rebuild → AUXIO regeneration. Full checklist: COLLECTION_ONBOARDING_BLUEPRINT.md.

Draft demo guide

        
Testing ARCHAI as an institutional toolkit

      

      
    

    

      

        
1. Search the collection
Use Converse for semantic questions across the database, Classic Search for direct matching, and image/colour modes as developing test lanes. Metadata-only records stay searchable, while public views require safer media and rights gates.

        
2. Review an object
Open any object and switch between Curatorial, Collections, Exhibition, and Interpretation lenses. These are staff-facing response modes, not visitor questions: they shape how ARCHAI reads the same record.

        
3. Build AUXIO pages
In AUXIO Management, select or create an object, assign it to a QR/NFC/spatial trigger, choose visibility options, preview the phone page, then publish. Production should save these edits in Directus or the institution's CMS/DAMS layer.

        
4. Use vocabulary tools
The thesaurus layer helps staff standardise terms, map local language to AAT/LCSH/TGN-style vocabularies, and flag culturally sensitive terminology before it becomes public interpretation.

        
5. Connect CMS and DAMS
The harvester layer is the next institutional-testing focus: custom connectors for APIs, CMS, DAMS, spreadsheets, IIIF manifests, and local media stores while preserving source records separately from interpretation.

        
6. Current boundaries
Nodel/exhibition ops and FAMTEC Exchange remain prototype modules. The strongest current lanes are collection search, object review, AUXIO publishing, rights/media gating, IIIF viewing, and browser voice access.
